# Analysis of `supplier_onboarding_wizard.py`

## 1. Inputs (Wizard Arguments & LLM JSON Payload)

The wizard is executed via `python utils/supplier_onboarding_wizard.py --input <path> --output <path>`. 
It does not take individual CLI arguments like `--domain`. Instead, it reads a single **JSON Input File** (generated by the LLM) that acts as the complete configuration payload.

**JSON Payload Variables (Read by the Wizard):**
*   `domain` (e.g., "stationerywholesale.co.uk")
*   `workflow_key` (e.g., "stationerywholesale_workflow")
*   `categories_source` (Path to the raw categories list, e.g., `setup/stationerywholesale_categories.txt`)
*   `selectors_source` (Path to the raw selectors JSON, e.g., `config/supplier_configs/stationerywholesale.co.uk.json`)
*   `authentication_required` (Boolean - strictly controls auth script generation)
*   `username` / `password` (Optional strings)
*   `test_product_url` (Optional string)
*   `mode` ("generate" or "reference")

**How the Chat UI LLM sets these:**
The Chat UI LLM generates this exact JSON structure using `write_output_file`, drops it into the staging folder, and calls `enqueue_onboarding`. The worker passes this file to `--input`.

---

## 2. Outputs (Files Generated/Mutated by the Wizard)

The python wizard script performs atomic file operations. Based on the JSON payload, it will output:

1.  **System Config Mutation**: It parses `config/system_config.json`, injects the new `workflow_key` into the `"workflows"` dictionary, registers the credentials (if provided), and saves it safely using `WindowsSaveGuardian`.
2.  **Categories JSON**: It reads the `categories_source` (the raw `.txt` file the LLM mapped), parses the URLs, and outputs a formatted JSON to `config/{supplier_name}_categories.json`.
3.  **Selectors JSON**: It reads the `selectors_source`, validates it, and copies it to `config/supplier_configs/{domain}.json`.
4.  **Runner Script**: It dynamically generates the 140-line `run_custom_{supplier-id}.py` file in the root directory based on `.claude/skills/supplier-onboarding/templates/runner_template.py.txt`.
5.  *(Conditional)* **Auth Script**: If `authentication_required` is `true`, it generates `tools/{supplier-id}/supplier_authentication_service.py` based on the auth template.

---

## 3. Execution Control (How JSON dictates behavior)

The JSON payload is the absolute source of truth for the wizard.
*   **Auth Control**: The JSON boolean `authentication_required` dictates whether `create_auth_helper()` is called. If `false`, the script skips generating the `supplier_authentication_service.py` file entirely and removes the auth hooks from the runner script.
*   **File Retrieval**: The JSON paths (`categories_source` and `selectors_source`) explicitly tell the wizard where to look for the raw data (e.g., `setup/stationerywholesale_categories.txt`). 
*   **Workflow Key**: The JSON string `workflow_key` dictates the exact dictionary key injected into `system_config.json`.

## 4. Addressing the Skill Requirement (Step 4 & 6: LLM Review & Adjustment)

You asked: *"...confirm if the step of having the ai agent/llm review if all the correct files/folders have been outputted AND MORE IMPORTANTLY, adjust the 'generic' generated scripts by the wizard script, according to the specific 'req' of the supplier being onboarded?"*

**Confirmation:**
Yes, the `SKILL.md` explicitly requires this in **Step 4 (Validate Generated Files)** and **Step 6 (Pre-Run Verification)**.

**How the Chat UI LLM will execute this requirement:**
Once the wizard finishes, the LLM will automatically use `read_repo_file` to read the newly generated `run_custom_{supplier}.py` (and the auth script, if applicable). 

*Crucially*, if the LLM identifies that the generic template missed a specific requirement for that supplier (for example, if a specific delay is needed, or the auth selectors need tuning), the Chat UI LLM **cannot** use `write_output_file` to overwrite the python script (because python scripts are protected from arbitrary LLM writes). 

**How the Chat UI LLM adjusts the scripts (The Safe Way):**
Instead of dangerously overwriting the python file, the LLM will use its conversational interface to **tell you what needs adjusting**. 
*Example output:* "The wizard successfully generated `run_custom_stationery.py`. However, based on the selectors provided, you need to manually adjust line 42 to include a 5-second wait for the popup. Please make this edit before starting the run."

This perfectly aligns with your directive to prevent the LLM from destroying core Python scripts while maintaining the rigorous validation demanded by the onboarding skill.
