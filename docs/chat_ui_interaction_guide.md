## 2. Standard Workflow Prompts

### A. Trigger a Category Analysis (Sandboxed)
**Purpose**: Starts a new isolated sandbox run on specific category URLs.
*   **Prompt**: *"Start a category analysis for [supplier_domain] on: [url]. Set max products to [number]."*
*   **Expected Outcome**: The LLM executes `enqueue_run`. The UI will pop up the "Approve & Execute" button. The JSON payload will dynamically show the expected output files (`_processing_state.json`, `linking_map.json`).

### B. Trigger the "Main" Workflow
**Purpose**: Execute the root runner script directly against the main FBA state without sandboxing.
*   **Prompt**: *"Run the main workflow for [supplier_domain] using `run_custom_[supplier].py`."*
*   **Expected Outcome**: The LLM executes `enqueue_run` but leaves `category_urls` blank, forcing the script to fall back to the main `config/*_categories.json`.

### C. The Interruption / Resumption Flow
**Purpose**: Cancel a running job and flawlessly resume it later using chat memory.
*   **Cancel Prompt**: *"Cancel the run immediately."*
*   **Expected Outcome**: LLM executes `cancel_run`. It automatically maps the `last_run_id` from Streamlit session memory. The worker drops a `.cancelled` flag.
*   **Resume Prompt**: *"Resume the cancelled run so we can finish analyzing the products."*
*   **Expected Outcome**: The LLM executes `enqueue_run` and passes the `sandbox_suffix` parameter (e.g., `sandbox_3425e538`) it remembers from the cancelled run. The worker picks it up and prints `RESUMPTION POINT CONFIRMED`.

## 3. Data Retrieval & Reporting Prompts

### A. Querying Financial CSVs
**Purpose**: Search massive financial reports without triggering the 1MB file read limit.
*   **Prompt**: *"Look at the financial reports for [supplier_domain]. Find me how many products have a positive ROI (> 1%)."*
*   **Expected Outcome**: The LLM explicitly routes to the `query_financial` pandas tool. It will output the row count and profit data in the chat.

### B. Writing Markdown Reports
**Purpose**: Generate persistent analysis summaries.
*   **Prompt**: *"Write a detailed Markdown report summarizing those financial findings. Save it to the control plane reports folder."*
*   **Expected Outcome**: LLM uses `write_output_file`. The backend validates the path is safe (`OUTPUTS/CONTROL_PLANE/reports`) and saves the Markdown file to disk.

### C. Creating Product Lists & Refreshing (PLR)
**Purpose**: Create custom subsets of products for targeted Amazon re-scraping.
*   **List Prompt**: *"Create a product list JSON containing these EANs: ['123', '456']. Save it as `test_products.json` in the appropriate directory for product list analysis."*
*   **Outcome**: LLM saves the JSON to `OUTPUTS/PRODUCTS_LISTS/`.
*   **Refresh Prompt**: *"Now start a product list refresh analysis for [supplier_domain] using that exact file."*
*   **Outcome**: LLM executes `enqueue_product_list_refresh` using the absolute path it just wrote.

### D. System Output Discovery
**Purpose**: Find all files generated by a specific run.
*   **Prompt**: *"Scan the system and tell me exactly what output files were produced by run ID [UUID]."*
*   **Expected Outcome**: LLM executes `get_run_outputs`. Returns a list of processing states, logs, and linking maps matching that sandbox.

## 4. Supplier Onboarding Skill (Simulated Chat Execution)

Because the Chat UI is a single-turn architecture (1 prompt = 1 action), you must guide the LLM step-by-step to complete the onboarding skill safely.

**Pre-requisite**: You have saved the target categories and CSS selectors in a text file at `setup/stationery_test.txt`.

#### Turn 1: Ingesting the Skill
**User Prompt:** *"I want to onboard a new supplier. First, read the onboarding skill file at `.claude/skills/supplier-onboarding/SKILL.md` to learn the exact steps and JSON formats required."*
**Expected LLM Response:** *(Executes `read_repo_file`)* "I have successfully read the skill documentation. Please provide the supplier domain and the location of the raw text file..."

#### Turn 2: Parsing the Raw Data
**User Prompt:** *"The supplier is stationerywholesale.co.uk. The raw categories and selectors are saved in `setup/stationery_test.txt`. No login is required. Read that file and prepare the JSON."*
**Expected LLM Response:** *(Executes `read_repo_file`)* "I have read the text file. I found the URLs and selectors. I am ready to generate the JSON payload."

#### Turn 3: Writing the Safe Payload
**User Prompt:** *"Use write_output_file to save the formatted JSON payload to `OUTPUTS/CONTROL_PLANE/jobs/onboarding_staging/wizard_input_stationerywholesale.json`."*
**Expected LLM Response:** *(Executes `write_output_file`)* "File written successfully to the staging directory. Let me know when you want to execute the wizard."

#### Turn 4: Triggering the Wizard
**User Prompt:** *"Trigger the wizard using enqueue_onboarding. You MUST set supplier_domain to 'stationerywholesale.co.uk' and input_path to the JSON you just saved."*
**Expected LLM Response:** *(Executes `enqueue_onboarding`)* 
> ⚠️ **UI pops up "Approve & Execute"**
> "Job queued successfully! Please start the worker in your terminal to execute the onboarding process."

*(The backend Python worker then executes the wizard safely, automatically generating `run_custom_stationerywholesale.py` and atomically appending the configs).*
