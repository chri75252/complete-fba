# 10.5. Dashboard Api

## Overview

The dashboard API (`dashboard_v2_redesign/api.py`) serves the React frontend dashboard. It aggregates metrics from all system output files and returns them in a single response.

**Base URL:** `http://127.0.0.1:8001`  
**Entry point:** `api.py` (FastAPI app)

---

## Endpoints

### `GET /api/metrics/{supplier}`

Primary endpoint. Returns all dashboard metrics for a given supplier.

**Parameters:**

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `supplier` | path | — | Supplier domain, e.g. `poundwholesale.co.uk` |
| `lineage` | query | `base` | `base` (main workflow) or `latest_sandbox` (most recent sandbox run) |
| `report` | query | `""` | Specific financial report filename. Empty = latest by mtime |
| `sales_field` | query | `bought_in_past_month` | `bought_in_past_month` or `amazon_sales_badge` |
| `force_refresh` | query | `false` | `1` = bypass all caches and recompute from scratch |

**Response fields:**

| Field | Source File | Dashboard Section |
|-------|------------|-------------------|
| `state_metrics` | `processing_state.json` | System Health: phase, category index, fresh starts, progress bars |
| `linking_metrics` | `linking_map.json` | Matching Performance: total matches, high confidence %, no EAN count, confidence distribution |
| `financial_metrics` | Latest CSV in `financial_reports/` | Top cards: profitable count, avg ROI, total profit, files scanned, charts |
| `supplier_cache_metrics` | `products_cache.json` | Total Extracted card (product count) |
| `amazon_cache_metrics` | `amazon_cache/` directory | Minor: file count and latest mtime |
| `log_data` | `run_custom_*.log` | Terminal Logs panel: last 200 lines |
| `data_sources` | — | Maps each section to the actual file path on disk |
| `paths` | — | Resolved file paths for the current supplier |

## Caching Architecture

Two independent caching layers protect against redundant file reads:

### Layer 1: Response Cache (`_response_cache`)

- **Location:** Module-level Python dict in `api.py`
- **Scope:** Entire JSON response (all sections combined)
- **Invalidation:** Time-only — 30 second TTL
- **Bypass:** `force_refresh=1` query parameter
- **Thread-safe:** Protected by `_cache_lock` (threading.Lock)

This layer is what makes repeated requests return in ~50ms instead of ~12s.

### Layer 2: MetricsLoader Internal Cache

- **Location:** `MetricsLoader._cache` dict inside the singleton loader instance
- **Scope:** Individual file parse results (one cache entry per file)
- **Invalidation:** Mtime-based — file modification time check on each read
- **Persistence:** Lives on the singleton loader instance (reused across all requests via `_get_loader()`)

This layer prevents re-reading and re-parsing a file within the same TTL window if the response cache has expired.

### Why Both Layers?

| Scenario | Response Cache Hit? | Loader Cache Hit? | Time |
|----------|-------------------|-------------------|------|
| Within 30s TTL | Yes (returns immediately) | N/A | ~50ms |
| After 30s, files unchanged | No (TTL expired) | Yes (mtime unchanged) | ~200ms |
| After 30s, files changed | No (TTL expired) | No (mtime changed) | ~12s |
| `force_refresh=1` | No (bypassed) | No (bypassed) | ~12s |
| Server restart | No (cache lost) | No (cache lost) | ~12s |

### Files That Are Never Cached

- **Log files** (`run_custom_*.log`) — always re-read from disk on every request because logs are continuously written during active workflow runs.

## Refresh Triggers

**What causes files to change (triggering re-read on next request after TTL expires):**

| File | Written By | When It Changes |
|------|-----------|----------------|
| `products_cache.json` | Supplier scraper | Every batch of new products scraped |
| `linking_map.json` | Amazon matching step | Every batch of products matched |
| `financial_reports/*.csv` | Financial calculator | After every N linking map entries processed |
| `processing_state.json` | Workflow engine | After every category batch completes |
| `run_custom_*.log` | Any runner script | Continuously during active runs |

**If no workflow is running:** none of these files change. After the first ~12s cold load, every request within 30s is ~50ms, and every request after 30s (TTL expired but files unchanged) is ~200ms.

## Auto-Refresh

The browser polls `/api/metrics` at the interval set in the Auto Refresh dropdown (default: 60 seconds).

| Dropdown Setting | Behavior |
|-----------------|----------|
| Disabled | No automatic polling. Click "Refresh Now" manually. |
| 3s / 5s | Polls every 3 or 5 seconds. Use only during active monitoring. |
| 30s | Good balance for watching a running workflow. |
| 60s | Default. Suitable for most monitoring scenarios. |
| 120s / 300s | Long intervals. Useful when data changes slowly. |

**Recommendation:** Set to **Disabled** when analyzing a completed run (no active workflow). The data is static and polling generates identical responses.

## Troubleshooting

### Dashboard shows stale data

1. Click **Refresh Now** — this bypasses both cache layers with `force_refresh=1`
2. Check if the Auto Refresh dropdown is set to a very long interval
3. Verify the source files exist at the paths shown in the Data Sources panel
4. If data is genuinely old (from a completed run weeks ago), this is expected — the files were not updated

### Metrics take 12+ seconds every time

1. First request after server start: expected, builds initial caches
2. Repeated requests taking 12s: `_response_cache` is not working. Check:
   - Is `force_refresh=1` being sent? (the Refresh Now button always sends it)
   - Is the server process crashing and restarting between requests?
3. Check `_CACHE_TTL_SECONDS` in `api.py` — if set to 0, cache is disabled

### Log panel not updating during active workflow

- Log files are never cached (intentional). If the Terminal Logs panel shows no new entries during a run:
  1. Check that the log file exists at the path shown in the Data Sources panel
  2. Verify the workflow runner is actually writing to that log file
  3. The log tail reads the last 200 lines only
